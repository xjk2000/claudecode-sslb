# 三省六部 Agent Teams

基于 Claude Code Agent Teams 的 **16 Agent** 协作系统，无外部依赖，开箱即用。

## 快速开始

### 安装（Claude Code Plugin）

```bash
cd ./claudecode-sslb
# 先安装marketplace
claude plugin marketplace add ./claudecode-sslb

# plugin 安装
/plugin install sslb@sslb
```

安装完成后，所有项目都可以使用三省六部 Agent。首次在项目中使用时，SessionStart hook 会自动创建 `docs/huangdi/` 目录结构。

卸载：`claude plugin uninstall sslb`

### 验证安装

在你的项目目录中打开 Claude Code，输入：
```
@中书令 你好
```
如果中书令回应，说明安装成功。

### 使用

```
# 方式一：Slash Command
/init-dynasty           # 初始化王朝（项目全局调研，建议首次使用时执行）
/new-edict              # 创建新敕令（完整流程）
/continue-edict         # 继续执行已有敕令
/debug                  # 启动调试
/review                 # 启动审查
/tdd                    # 启动 TDD

# 方式二：@Agent 直接调用
@zhongshuling 分析用户注册需求
@shangshuling 实现用户注册功能    # 尚书令自动判断需要哪些 Agent 协作
@bingbu 为 UserService 写测试
@xingbu 登录接口返回500
@gongbu review 这次提交
```

## 系统架构

```
用户（皇帝）
    │
    ▼
中书省 ─ 中书令 + 中书舍人
    │     需求探索 → 敕令拆解 → 编号分发
    ▼
门下省 ─ 侍中 + 给事中
    │     封驳审议 → 放行/打回
    ▼
尚书省 ─ 尚书令（判断协作Agent → 制定计划 → dispatch）
    │
    ├─ 吏部 ── Agent能力匹配建议
    ├─ 户部 ── 数据收集分析
    ├─ 礼部 ── 文档编写审核
    ├─ 兵部 ── TDD 测试 → 打回五监
    ├─ 刑部 ── 系统调试 → 打回五监
    ├─ 工部 ── 代码审查 → 打回五监
    │
    └─ 五监（代码实现，按逻辑领域路由）
       ├─ 将作监 ── 核心业务逻辑
       ├─ 少府监 ── 前端交互
       ├─ 军器监 ── 安全认证
       ├─ 都水监 ── 数据处理
       └─ 国子监 ── 框架架构
```

## 核心流程

```
用户指令
    │
    ▼
中书令需求探索 ──→ 中书舍人文档化 ──→ 侍中(+给事中)审议
                                              │
                                          封驳？─→ 打回中书令
                                              │
                                            放行
                                              │
                                              ▼
                                        尚书令制定计划
                                    (判断需要哪些Agent协作)
                                              │
                                     dispatch subagent
                                ┌─────────────┼─────────────┐
                                │             │             │
                              兵部          五监          户部/礼部
                              (TDD)       (实现)        (数据/文档)
                                │             │
                                └──打回五监──┘
                                       │
                                   工部 review
                                       │
                                   侍中验收
                                       │
                                   中书令汇报
```

## 四大独有机制

### 1. 敕令制度
所有任务以「敕令」形式流转，有编号、优先级、验收标准。

| 类型 | 编号格式 | 说明 |
|------|----------|------|
| 中书敕令 | ZS-YYYYMMDD-XXX | 正式任务敕令 |
| 兵部打回 | BTD-YYYYMMDD-XXX | 测试失败打回 |
| 刑部打回 | TIAO-YYYYMMDD-XXX | Bug诊断打回 |
| 工部打回 | GBR-YYYYMMDD-XXX | 审查问题打回 |

### 2. 封驳机制
门下省侍中审议敕令，不合格则封驳打回中书令修改。给事中辅助做技术细节审查和风险评估。

### 3. 打回机制
**发现问题的人不修代码，修代码的人不做诊断。**

| 场景 | 流向 |
|------|------|
| 兵部写失败测试 | → 打回五监实现代码 |
| 刑部定位根因 | → 打回五监修复 |
| 工部发现问题 | → 打回五监改正 |

### 4. 过程记录（政事堂 / 弘文馆）
所有敕令的完整生命周期记录在 `docs/huangdi/` 目录中：

```
docs/huangdi/
├── 政事堂/                当前活跃敕令
│   ├── 诏-20260327-001/    敕令文件夹（含主文件+计划+打回记录）
│   └── 诏-20260327-002/
├── 秘书省/                归档敕令文件夹
├── 弘文馆/                敕令总结库
└── TEMPLATE-edict.md        敕令模板
```

- **政事堂**：每个敕令一个文件夹（`诏-YYYYMMDD-XXX/`），内含敕令主文件、执行计划、打回记录等全过程文档
- **秘书省**：验收通过后，整个敕令文件夹移入此处归档
- **弘文馆**：敕令总结存入此处，作为知识库供后续参考
- **防漂移**：每次会话启动时先扫描政事堂中的 `诏-*/` 文件夹，恢复上下文

## 尚书令的 Agent 协作判断

尚书令收到敕令后，**首先**分析需要哪些 Agent 参与。示例：

```
敕令: "实现用户注册功能"
→ 协作Agent: 军器监(认证) + 都水监(数据库) + 将作监(业务) + 兵部(TDD) + 工部(review) + 礼部(API文档)

敕令: "修复登录页面样式错乱"
→ 协作Agent: 刑部(诊断) + 少府监(前端修复) + 兵部(验证) + 工部(review)

敕令: "优化数据库查询性能"
→ 协作Agent: 户部(性能分析) + 都水监(SQL优化) + 兵部(性能测试) + 工部(review)
```
