# 敕令拆解技能

将确认后的设计方案拆解为标准化敕令，并做出路由决策。

## 何时使用

- 中书令完成 brainstorming 并获得用户确认后
- 需要将一个大任务分解为多个可执行敕令时

## 拆解原则

1. **单一职责** — 每个敕令只做一件事
2. **可验证** — 每个敕令有明确的验收标准
3. **可路由** — 每个敕令有唯一的主责部门
4. **无歧义** — 五监可以"无脑执行"

## 拆解流程

### Step 1: 识别任务边界

从设计文档中提取独立的工作单元：
- 每个独立的功能模块 → 一个敕令
- 每个独立的 Bug 修复 → 一个敕令
- 跨模块的集成工作 → 一个独立的集成敕令

### Step 2: 生成敕令

为每个工作单元生成标准化敕令：

```markdown
# 中书敕令

**敕令编号**: ZS-YYYYMMDD-XXX
**优先级**: P0/P1/P2/P3
**类型**: feature/bugfix/refactor/review/docs

## 任务概述
[一句话描述，20字以内]

## 执行部门
**主责**: [部门名]
**协作**: [部门名（如有）]

## 详细要求
[具体要做什么，分步骤列出]

## 验收标准
1. [具体可验证的标准1]
2. [具体可验证的标准2]

## 依赖关系
- 前置: [依赖的敕令编号（如有）]
- 后续: [被依赖的敕令编号（如有）]

## 关联设计
[docs/specs/xxx-design.md]
```

### Step 3: 路由决策

根据任务类型和代码归属确定路由：

| 任务特征 | 路由到 |
|----------|--------|
| 新功能实现 | 尚书令（制定计划后 dispatch 五监） |
| Bug 修复 | 刑部（诊断后打回五监） |
| 需要先写测试 | 兵部（写测试后打回五监） |
| 代码审查 | 工部 |
| 复合任务 | 尚书令（二次拆解） |

### Step 4: 排序和依赖

确定敕令执行顺序：
1. 无依赖的敕令优先
2. 基础设施敕令先于业务敕令
3. 有依赖关系的按依赖顺序排列

## 输出格式

```markdown
# 敕令清单

**设计文档**: docs/specs/xxx-design.md
**总敕令数**: N
**预估工时**: [总工时]

## 执行顺序

### Phase 1: 基础设施
1. ZS-YYYYMMDD-001 — [描述] → 尚书令
2. ZS-YYYYMMDD-002 — [描述] → 尚书令

### Phase 2: 核心功能
3. ZS-YYYYMMDD-003 — [描述] → 尚书令（依赖 001）
4. ZS-YYYYMMDD-004 — [描述] → 兵部

### Phase 3: 集成验证
5. ZS-YYYYMMDD-005 — [描述] → 工部
```

## 质量检查

生成敕令后自检：
- [ ] 每个敕令都有明确的主责部门？
- [ ] 每个敕令的验收标准都是可验证的？
- [ ] 依赖关系是否形成环？（不允许循环依赖）
- [ ] 所有设计文档中的需求都被覆盖了？
- [ ] 没有遗漏的需求项？
