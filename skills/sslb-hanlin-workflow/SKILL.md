# 翰林学士工作流

独立于三省六部的单 agent 全流程开发工作流。直接接收用户指令，直接执行。

## 核心铁律

```
1. 动手前先想清楚（Brainstorming）
2. 想清楚再写计划（Planning）
3. 写测试再写代码（TDD）
4. 证据先于断言（Verification）
5. 根因先于修复（Debugging）
```

## Phase 1: Brainstorming（需求澄清）

**触发条件**：用户提出新任务、新功能、新需求。

**流程**：
1. 检查项目现状（文件、文档、最近提交）
2. **一次只问一个问题**，优先选择题
3. 提出 2-3 种方案 + 权衡 + 你的推荐
4. 分段展示设计，每段获得用户确认
5. 写设计文档到 `我的帝国朝堂/翰林院/specs/YYYY-MM-DD-<topic>.md`
6. 自审：检查占位符、矛盾、模糊点，修复后让用户确认
7. 用户确认后进入 Planning

**硬门禁**：设计未获用户批准前，**禁止写任何代码**。

**关键原则**：
- YAGNI — 不需要的就砍掉
- 一次一个问题 — 不要一次抛出 5 个问题轰炸用户
- 简单任务也需要设计 — 哪怕只用几句话

## Phase 2: Planning（实现计划）

**触发条件**：设计获批后。

**流程**：
1. 将设计拆解为 bite-sized 任务（每个 2-5 分钟）
2. 每个任务必须包含：精确文件路径、完整代码、验证命令
3. 写计划到 `我的帝国朝堂/翰林院/plans/YYYY-MM-DD-<feature>.md`
4. 自审：spec 覆盖度、占位符扫描、类型一致性
5. 用 TodoWrite 创建任务清单

**任务粒度**：
```
- "写失败测试" — 一步
- "运行确认失败" — 一步
- "写最小实现通过测试" — 一步
- "运行确认通过" — 一步
- "提交" — 一步
```

**禁止占位符**：
- ❌ "TBD"、"TODO"、"后面补充"
- ❌ "添加适当的错误处理"
- ❌ "类似任务 N"（必须重复写出完整代码）

## Phase 3: TDD（测试驱动开发）

**铁律**：`没有失败测试，就不写生产代码`

**RED → GREEN → REFACTOR 循环**：

### RED — 写失败测试
- 一个测试只测一个行为
- 名称即文档：`test('拒绝空邮箱')` 而非 `test('test1')`
- 用真实代码，不 mock（除非不可避免）

### 验证 RED — 看它失败
```bash
# 运行测试，确认失败
[项目测试命令]
```
确认：失败原因是"功能未实现"而非"代码有 typo"。

### GREEN — 最小实现
写**刚好够**通过测试的代码。不多写。

### 验证 GREEN — 看它通过
```bash
[项目测试命令]
```
确认：目标测试通过 + 其他测试未被破坏 + 输出干净。

### REFACTOR — 清理
消除重复、改进命名、提取辅助方法。保持测试绿色。

**违反铁律的处理**：先写了代码再补测试？**删掉代码，从测试开始重来。**

## Phase 4: Systematic Debugging（系统化调试）

**铁律**：`没有根因调查，就不提修复方案`

**四阶段流程**：

### 4.1 根因调查
1. 仔细阅读错误信息和完整堆栈
2. 稳定复现（精确步骤、是否每次发生）
3. 检查最近变更（git diff、新依赖、配置变更）
4. 追踪数据流（坏值从哪来→谁传进来→继续往上追→在源头修复）

### 4.2 模式分析
1. 找到同项目中类似的正常代码
2. 逐项对比差异
3. 理解依赖关系

### 4.3 假设与测试
1. 形成单一假设："我认为 X 是根因，因为 Y"
2. 最小化验证 — 一次只改一个变量
3. 有效→Phase 4.4；无效→新假设
4. **3 次修复尝试都失败 → 停止，向用户汇报，可能是架构问题**

### 4.4 实现修复
1. 先写失败测试复现 Bug
2. 实现单一修复（只改根因）
3. 验证修复 + 无回归

## Phase 5: Verification（完成验证）

**铁律**：`没有运行验证命令，就不宣称完成`

**每次声称完成前**：
1. **识别**：什么命令能证明这个声明？
2. **运行**：执行完整命令
3. **阅读**：完整输出、检查退出码
4. **确认**：输出是否支持声明？
5. **然后才能声明**

**禁止**：
- ❌ "应该可以了"、"大概没问题"
- ❌ 没运行测试就说测试通过
- ❌ 没运行构建就说构建成功

## 记忆管理

翰林学士的记忆存放于 `我的帝国朝堂/翰林院/`：

```
我的帝国朝堂/翰林院/
├── specs/          # 设计文档
├── plans/          # 实现计划
└── records/        # 任务记录
```

每次任务完成后，记录到 `records/YYYY-MM-DD-<task>.md`。

## 红线（出现以下想法时立即停下）

| 想法 | 现实 |
|------|------|
| "太简单了不用设计" | 简单任务是未审视的假设浪费最多工作的地方 |
| "先写代码再补测试" | 后补的测试立即通过，证明不了任何事 |
| "快速修一下试试" | 猜测修复浪费时间，系统调试更快 |
| "应该可以了" | 运行验证命令，用证据说话 |
| "这次先跳过流程" | 没有例外 |
