# 打回派发技能

三省六部独有的打回机制：当兵部测试失败或刑部定位根因后，将问题正式打回给对应的五监修复。

## 何时使用

- 兵部写了失败测试，需要五监实现代码
- 刑部定位了 Bug 根因，需要五监修复
- 工部审查发现问题，需要五监改正

## 核心原则

1. **职责隔离** — 发现问题的人不修复问题
2. **信息完整** — 打回敕令必须包含足够信息让五监"无脑修复"
3. **可验证** — 打回敕令必须包含验证命令
4. **可追踪** — 每次打回有编号，可追溯

## 打回流程

### Step 1: 确定责任五监

根据问题代码的逻辑领域确定路由（具体代码位置因项目而异）：

| 逻辑领域 | 路由到 |
|----------|--------|
| 业务规则、领域模型、业务服务 | 将作监 |
| UI组件、页面、交互、样式 | 少府监 |
| 认证、授权、加密、安全 | 军器监 |
| 数据库、数据转换、数据校验 | 都水监 |
| 框架、中间件、配置、基础设施 | 国子监 |

**跨领域问题**：如果涉及多个领域，以根因所在领域为主。

### Step 2: 生成打回敕令

#### 兵部打回（测试失败）

```markdown
# 兵部打回敕令

**编号**: BTD-YYYYMMDD-XXX
**来源**: 兵部
**主送**: [五监角色]
**源敕令**: ZS-YYYYMMDD-XXX
**紧急度**: [P0-P3]

## 失败测试
- **文件**: `tests/path/to/test.test.ts`
- **测试名**: `[具体测试函数名]`
- **失败类型**: 功能未实现 / 逻辑错误 / 边界条件

## 失败输出
```
[完整的测试失败输出，包含错误信息和堆栈]
```

## 期望行为
[测试期望代码做什么]

## 当前行为
[代码实际做了什么（或函数不存在）]

## 修复要求
[明确的修复要求]

## 涉及文件
- `src/path/to/file.ts` — [需要做什么]

## 验证命令
```bash
npm test tests/path/to/test.test.ts
```

## 验证标准
- [ ] 上述测试通过
- [ ] 其他测试未被破坏
```

#### 刑部打回（Bug 修复）

```markdown
# 刑部调试报告 & 打回敕令

**编号**: TIAO-YYYYMMDD-XXX
**来源**: 刑部
**主送**: [五监角色]
**源敕令**: ZS-YYYYMMDD-XXX
**紧急度**: [P0-P3]

## 症状
[用户观察到的异常行为]

## 根因分析
- **位置**: `src/path/to/file.ts:行号`
- **原因**: [根因描述]
- **证据**: [诊断过程中收集的证据]

## 修复方向
[建议的修复方向，但不指定具体代码]

## 涉及文件
- `src/path/to/file.ts:行号` — [需要修改什么]

## 回归测试
[需要通过的已有测试]

## 验证命令
```bash
[验证命令]
```

## 验证标准
- [ ] 原始 Bug 不再复现
- [ ] 回归测试全部通过
- [ ] 无新增 warning/error
```

#### 工部打回（审查修复）

```markdown
# 工部审查反馈

**编号**: GBR-YYYYMMDD-XXX
**来源**: 工部
**主送**: [五监角色]
**源敕令**: ZS-YYYYMMDD-XXX

## 审查发现

### 🔴 Critical（必须修复）
1. **`file.ts:行号`**: [问题] → [修复建议]

### 🟠 Important（应该修复）
1. **`file.ts:行号`**: [问题] → [修复建议]

## 验证标准
- [ ] 所有 Critical 问题已修复
- [ ] 所有 Important 问题已修复或说明原因
- [ ] 测试通过
```

### Step 3: Dispatch 五监

将打回敕令 dispatch 给对应的五监 subagent：

1. 附上打回敕令全文
2. 附上相关的上下文（关联文件内容）
3. 等待五监完成修复
4. 收到五监回复后，进入验证环节

### Step 4: 验证修复

五监修复后，由打回方（兵部/刑部/工部）执行验证：

- **兵部打回** → 兵部重新运行测试
- **刑部打回** → 刑部验证 Bug 不再复现
- **工部打回** → 工部重新审查

**验证不通过 → 重新打回（附上新的失败信息）**

## 打回计数

每个敕令维护打回计数：

| 打回次数 | 处理 |
|----------|------|
| 1-2 次 | 正常打回修复循环 |
| 3 次 | 警告，可能需要重新审视方案 |
| 5+ 次 | 暂停，上报尚书令/中书令，可能是设计问题 |

## 编号规则

| 类型 | 前缀 | 格式 |
|------|------|------|
| 兵部打回 | BTD | BTD-YYYYMMDD-XXX |
| 刑部打回 | TIAO | TIAO-YYYYMMDD-XXX |
| 工部打回 | GBR | GBR-YYYYMMDD-XXX |

其中 XXX 为当日序号，从 001 开始递增。
