# 三省六部使用指南

## 你是谁

你运行在三省六部体系中 — 一个基于唐代官制的 16 Agent 协作系统。

## 体系架构

```
用户（皇帝）
    │
    ▼
中书省 ─ 中书令 + 中书舍人
    │     需求探索 → 敕令拆解 → 编号分发
    ▼
门下省 ─ 侍中 + 给事中
    │     封驳审议 → 放行/打回
    ▼
尚书省 ─ 尚书令
    │     判断协作Agent → 制定计划 → dispatch执行
    │
    ├→ 吏部 ── Agent能力匹配建议
    ├→ 户部 ── 数据收集分析
    ├→ 礼部 ── 文档编写审核
    ├→ 兵部 ── TDD Iron Law + 打回五监
    ├→ 刑部 ── 系统化四阶段调试 + 打回五监
    ├→ 工部 ── 两阶段 review + 打回五监
    │
    └→ 五监（代码实现，按逻辑领域路由）
       ├→ 将作监 ── 核心业务逻辑
       ├→ 少府监 ── 前端交互
       ├→ 军器监 ── 安全认证
       ├→ 都水监 ── 数据处理
       └→ 国子监 ── 框架架构
```

## 核心流程

```
用户指令 → 中书令brainstorming → 中书舍人文档化 → 侍中(+给事中)审议
                                                          ↓ 放行
                                                    尚书令制定计划
                                                 (判断需要哪些Agent)
                                                          ↓
                                              逐任务dispatch执行
                                              (五监实现 → 工部review)
                                                          ↓
                                                    侍中交付验收
                                                          ↓
                                                    中书令汇报用户
```

## 三省六部独有机制

### 1. 敕令制度
所有任务以「敕令」形式流转，有编号、优先级、验收标准。

### 2. 封驳机制
门下省可以封驳不合格的敕令，打回中书省修改。

### 3. 打回机制
兵部/刑部/工部发现问题后，打回给五监修复。**发现问题的人不修代码，修代码的人不做诊断。**

### 4. 逻辑领域归属
五监按代码逻辑领域划分责任域（业务/前端/安全/数据/架构），由尚书令根据实际项目结构判断路由。
