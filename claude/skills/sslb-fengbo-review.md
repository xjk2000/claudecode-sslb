# 封驳审议技能

门下省侍中的核心技能：对中书令生成的敕令进行审议，决定放行、封驳或要求修改。

## 何时使用

- 中书令生成新敕令后提交审议时
- 敕令修改后重新提交审议时

## 审议流程

### Step 1: 敕令完整性检查

逐项检查敕令格式是否完整：

- [ ] 有敕令编号？
- [ ] 有优先级？
- [ ] 有任务概述（20字以内）？
- [ ] 有明确的主责部门？
- [ ] 有可验证的验收标准？
- [ ] 有关联的设计文档？

**任何一项缺失 → 直接打回中书令补充，不进入后续审议。**

### Step 2: 风险评估

评估以下风险维度：

| 维度 | 低风险 | 高风险 |
|------|--------|--------|
| **数据安全** | 只读操作 | 写入/删除/修改数据 |
| **可回滚性** | 可轻松回滚 | 不可回滚或回滚成本高 |
| **影响范围** | 单模块 | 跨多模块/全局 |
| **依赖变更** | 无新依赖 | 引入新外部依赖 |
| **API 变更** | 向后兼容 | Breaking change |

**高风险项 ≥ 2 → 要求额外说明或分步执行方案。**

### Step 3: 技术可行性

- 依赖的 API/库是否存在？
- 技术方案是否在项目技术栈范围内？
- 是否有已知的技术障碍？

### Step 4: YAGNI 检查

- 是否有不需要的功能？
- 是否过度设计？
- 是否做了用户没要求的事？

**违反 YAGNI → 封驳，要求精简。**

### Step 5: 做出决定

#### 放行条件（全部满足）

```
✅ 格式完整
✅ 风险可控（或有风险缓解方案）
✅ 技术可行
✅ 不违反 YAGNI
✅ 验收标准可量化
```

#### 封驳/修改

```markdown
# 门下封驳

**原敕令**: ZS-YYYYMMDD-XXX
**决定**: 封驳 / 要求修改
**审议官**: 侍中

## 原因
[具体理由，引用审议步骤中的发现]

## 问题项
1. [问题1]
2. [问题2]

## 修改建议
[如何改进才能通过审议]
```

#### 放行

```markdown
# 门下放行

**原敕令**: ZS-YYYYMMDD-XXX
**决定**: 放行
**审议官**: 侍中

## 审议摘要
- 风险等级: 低/中/高
- 注意事项: [如有]

## 放行条件
[如有额外的执行期注意事项]
```

## 审议速度要求

- **P0 敕令**: 跳过前置审议，但执行后仍需验收
- **P1 敕令**: 2 分钟内完成审议
- **P2-P3 敕令**: 正常审议流程

## 常见封驳理由

| 理由 | 示例 |
|------|------|
| 验收标准模糊 | "让它工作得更好" → 必须量化 |
| 过度设计 | 简单 CRUD 却要引入消息队列 |
| 缺少回滚方案 | 数据迁移没有回滚脚本 |
| 依赖不存在 | 引用了不存在的 API |
| 与已有敕令冲突 | 两个敕令修改同一模块但方向矛盾 |
