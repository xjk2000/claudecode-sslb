# 工部代码审查技能

## 概述

审查代码变更，确保质量合规。

**核心原则**: 及时审查，经常审查。

**五监反馈机制**: 工部发现问题后，反馈给五监修复。

## 何时请求审查

**必须审查**:
- 每次任务完成后
- 主要功能实现完成后
- 合并到主分支前

**可选但有价值**:
- 卡住时（新鲜视角）
- 重构前（基线检查）
- 复杂bug修复后

## 审查范围

### 代码质量
- [ ] **可读性**：变量/函数命名清晰
- [ ] **简洁性**：避免不必要的复杂性
- [ ] **一致性**：遵循项目规范
- [ ] **模块化**：职责单一，接口清晰

### 逻辑正确性
- [ ] **边界条件**：空值、零值、极值
- [ ] **错误处理**：异常有兜底
- [ ] **并发安全**：正确处理多线程

### 性能考量
- [ ] **算法复杂度**：避免O(n²)
- [ ] **数据库查询**：无N+1问题
- [ ] **懒加载**：按需加载

### 安全检查
- [ ] **输入验证**：外部输入验证
- [ ] **SQL注入**：参数化查询
- [ ] **敏感数据**：不硬编码密钥

## 问题严重程度

| 级别 | 标记 | 说明 | 处理 |
|------|------|------|------|
| 阻塞 | 🔴 | 必须修复才能合并 | 阻塞 |
| 严重 | 🟠 | 强烈建议修复 | 讨论 |
| 轻微 | 🟡 | 建议改进 | 后续 |
| 赞誉 | 🟢 | 优秀实践 | 表扬 |

## 五监反馈机制

发现代码问题后：

1. **记录问题**
2. **明确责任五监**
3. **发送反馈给对应五监**
4. **等待修复**
5. **验证修复**
6. **批准或继续反馈**

## 五监归责

| 问题位置 | 责任五监 |
|----------|----------|
| src/core/* | 将作监 |
| src/frontend/* | 少府监 |
| src/security/* | 军器监 |
| src/data/* | 都水监 |
| src/framework/* | 国子监 |

## 最佳实践

### 给开发者的建议
1. **保持变更小巧**：每次PR少于400行
2. **自审查**：提交前自己先审查
3. **写好描述**：说明Why而非What
4. **响应及时**：尽快处理反馈

### 给审查者的建议
1. **及时审查**：24小时内完成
2. **建设性反馈**：指出问题时提供方案
3. **区分主次**：阻塞问题明确标注
4. **考虑情境**：理解业务背景

## 验证清单

审查前检查:
- [ ] 理解变更目的和背景
- [ ] 运行本地测试确保通过
- [ ] 检查变更范围

审查时检查:
- [ ] 代码逻辑正确
- [ ] 边界条件覆盖
- [ ] 错误处理完善
- [ ] 性能合理
- [ ] 安全性合规
- [ ] 测试覆盖充分
