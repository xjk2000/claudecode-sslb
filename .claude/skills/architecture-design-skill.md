# 架构设计技能
国子监（Imperial Academy）执行的标准架构设计流程。

## 核心职责

- 系统架构设计
- 技术选型决策
- 框架与模式定义
- 技术债务管理
- 架构评审主持

## 架构设计原则

### SOLID 原则
```
┌─────────────────────────────────────────────────────────┐
│                    SOLID 原则                           │
├─────────────┬───────────────────────────────────────────┤
│ S           │ Single Responsibility (单一职责)         │
│ O           │ Open/Closed (开闭原则)                    │
│ L           │ Liskov Substitution (里氏替换)           │
│ I           │ Interface Segregation (接口隔离)          │
│ D           │ Dependency Inversion (依赖反转)           │
└─────────────┴───────────────────────────────────────────┘
```

### DRY / KISS / YAGNI
- **DRY**: Don't Repeat Yourself - 避免重复
- **KISS**: Keep It Simple, Stupid - 保持简单
- **YAGNI**: You Aren't Gonna Need It - 不要过度设计

## 架构设计流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  REQUIRE    │───►│    DESIGN   │───►│   REVIEW    │───►│   DOCUMENT   │
│   需求      │    │   设计      │    │   评审      │    │   文档化    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### Phase 1: 需求分析
1. 理解业务目标
2. 收集功能需求
3. 识别非功能需求
4. 定义约束条件

### Phase 2: 架构设计
1. 确定架构风格
2. 划分系统模块
3. 定义接口契约
4. 选择技术栈

### Phase 3: 架构评审
1. 组织评审会议
2. 评估设计方案
3. 识别风险点
4. 给出评审结论

### Phase 4: 文档化
1. 编写架构文档
2. 绘制架构图
3. 记录决策理由
4. 维护设计规范

## 技术选型指南

### 评估维度

| 维度 | 权重 | 评估项 |
|------|------|--------|
| 成熟度 | 25% | 社区活跃度、版本稳定性、生产验证 |
| 性能 | 20% | 响应时间、吞吐量、资源消耗 |
| 生态 | 15% | 周边工具、文档质量、学习曲线 |
| 风险 | 20% | 供应商锁定、安全漏洞、维护成本 |
| 成本 | 20% | 许可费用、人力成本、基础设施 |

### 技术选型决策表
```markdown
## 技术选型评估表

| 技术 | 成熟度 | 性能 | 生态 | 风险 | 成本 | 总分 |
|------|--------|------|------|------|------|------|
| A    | 4/5    | 5/5  | 4/5  | 4/5  | 3/5  | 4.0  |
| B    | 5/5    | 3/5  | 5/5  | 5/5  | 4/5  | 4.4  |
| C    | 3/5    | 4/5  | 3/5  | 3/5  | 5/5  | 3.6  |

### 结论
推荐选择 **B** 技术，综合得分最高，且风险最低。
```

## 架构评审检查清单

### 设计评审
- [ ] 是否符合业务需求？
- [ ] 是否满足非功能需求？
- [ ] 是否考虑了扩展性？
- [ ] 是否有技术风险？
- [ ] 是否有替代方案对比？

### 代码评审
- [ ] 模块划分是否清晰？
- [ ] 接口设计是否合理？
- [ ] 是否符合设计原则？
- [ ] 是否有明显的坏味道？

### 基础设施评审
- [ ] 部署复杂度如何？
- [ ] 运维成本如何？
- [ ] 监控告警是否完善？
- [ ] 灾备方案是否考虑？

## 技术债务管理

### 债务识别
```markdown
## 技术债务登记

| ID | 描述 | 影响 | 修复成本 | 优先级 |
|----|------|------|----------|--------|
| TD-001 | 硬编码配置 | 高 | 2d | P1 |
| TD-002 | 缺少单元测试 | 中 | 5d | P2 |
```

### 债务偿还策略
1. **持续改进**: 每次修改相关代码时顺便重构
2. **专门迭代**: 安排专项迭代偿还债务
3. **重大重构**: 必要时进行架构升级
