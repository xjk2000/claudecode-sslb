# 刑部系统化调试技能

## 概述

随机修复浪费时间并产生新bug。快速补丁掩盖根本问题。

**核心原则**: 在尝试修复前总是找到根本原因。症状修复是失败。

**铁律**: 没有根本原因调查就不进行修复。

**五监打回机制**: 刑部只负责调试诊断发现问题，分析完成后打回给五监修复。

## 五监职责划分

```
┌─────────────────────────────────────────────────────────────────┐
│                         三省六部职责                              │
├─────────────────────────────────────────────────────────────────┤
│  刑部（调试部）                                                 │
│  ├── 接收尚书令/中书省调试任务                                   │
│  ├── 收集症状和上下文                                          │
│  ├── 定位根因（不问责任）                                       │
│  ├── 分析问题归属哪个五监部门                                   │
│  └── 分析完成 → 打回给五监修复                                  │
├─────────────────────────────────────────────────────────────────┤
│  五监（修复部）                                                 │
│  ├── 将作监：核心业务逻辑bug                                    │
│  ├── 少府监：前端交互bug                                        │
│  ├── 军器监：安全相关bug                                        │
│  ├── 都水监：数据处理bug                                        │
│  └── 国子监：框架/架构bug                                       │
└─────────────────────────────────────────────────────────────────┘
```

## 何时使用

对任何技术问题使用:
- 测试失败
- 生产bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在以下情况使用**:
- 时间压力大时（紧急情况使人容易猜测）
- "就一个快速修复"看起来很明显时
- 你已经尝试了多个修复
- 之前的修复没有起作用
- 你不完全理解问题时

## 四阶段流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  阶段一    │───►│  阶段二    │───►│  阶段三    │───►│  阶段四    │
│  根因调查  │    │  模式分析  │    │  假设验证  │    │  实施修复  │
└─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
                                                                   │
                                                                   ▼
                                                            ┌─────────────┐
                                                            │  打回五监   │
                                                            │  修复       │
                                                            └─────────────┘
```

### 阶段一：根因调查

**在尝试任何修复之前**：

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪

2. **一致地复现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变更可能导致这个？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **追踪数据流**
   - 错误值从哪里来？
   - 谁调用了它并传入了坏值？
   - 一直追踪到找到源头
   - 在源头修复，不是症状

### 阶段二：模式分析

**在修复之前找到模式**：

1. **找到工作示例**
   - 在同一代码库中找到类似的工作代码
   - 什么能工作且类似失败的？

2. **与参考对比**
   - 如果是实现模式，完整阅读参考实现
   - 不要略读 - 阅读每一行
   - 在应用前完全理解模式

3. **识别差异**
   - 工作的和失败的区别是什么？
   - 列出每个差异，无论多小
   - 不要假设"那不可能有关系"

4. **理解依赖**
   - 这个需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

### 阶段三：假设和测试

**科学方法**：

1. **形成单一假设**
   - 清楚地陈述："我认为X是根本原因，因为Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小的可能变更来测试假设
   - 一次一个变量
   - 不要一次修复多个

3. **验证后再继续**
   - 有效？→ 阶段四
   - 无效？→ 形成新假设
   - 不要在顶部添加更多修复

### 阶段四：实施修复

**修复根本原因，不是症状**：

1. **创建失败测试用例**
   - 最简单的可能复现
   - 如果可能，自动化测试

2. **实施单一修复**
   - 解决已识别的根本原因
   - 一次一个变更
   - 不要"顺手"改进

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试被破坏？
   - 问题真的解决了？

4. **如果3+修复失败：质疑架构**
   - 每个修复都揭示了不同地方的新问题？
   - 修复需要"大规模重构"才能实施？
   - 每个修复在其他地方产生了新症状？

### 调试完成后的打回流程

当刑部完成根因分析后：

1. **填写调试报告 & 打回敕令**
2. **确定责任五监**
3. **发送打回敕令**
4. **等待五监修复**
5. **验证修复**

## 红旗 - 停止并遵循流程

如果发现自己有这些想法:
- "先快速修复，以后再调查"
- "就试试改X看看是否有效"
- "添加多个变更，运行测试"
- "跳过测试，我手动验证"
- "可能是X，让我修复那个"
- 在追踪数据流之前提出解决方案
- **"再试一次修复"（已经尝试了2+次）**
- **每个修复都在不同地方揭示了新问题**

**所有这些意味着：停止。回到阶段一。**
