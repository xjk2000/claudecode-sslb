# 系统化调试技能
刑部（Debugging Ministry）执行的标准调试流程。

**核心原则**：刑部只负责调试诊断发现问题，发现根因后打回给五监修复。

## 职责划分

```
┌─────────────────────────────────────────────────────────────────┐
│                         三省六部职责                              │
├─────────────────────────────────────────────────────────────────┤
│  刑部（调试部）                                                 │
│  ├── 接收尚书令/中书省调试任务                                   │
│  ├── 收集症状和上下文                                          │
│  ├── 定位根因（不问责任）                                       │
│  ├── 分析问题归属哪个五监部门                                   │
│  └── 分析完成后 → 打回给五监修复                                 │
├─────────────────────────────────────────────────────────────────┤
│  五监（修复部）                                                 │
│  ├── 将作监：核心业务逻辑bug                                    │
│  ├── 少府监：前端交互bug                                        │
│  ├── 军器监：安全相关bug                                        │
│  ├── 都水监：数据处理bug                                        │
│  └── 国子监：框架/架构bug                                       │
└─────────────────────────────────────────────────────────────────┘
```

## 四阶段调试流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  收集症状   │───►│  定位根因   │───►│  分析归责   │───►│  打回五监   │
│   症状      │    │   根因      │    │   归责      │    │   修复      │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### Phase 2: 定位根因

#### 四问法定位根因
```
┌─────────────────────────────────────────────────────────────────┐
│                        四问法                                    │
├─────────────────────────────────────────────────────────────────┤
│  1. 这个bug发生在哪里？    → 定位文件和行号                       │
│  2. 为什么会在那里发生？  → 分析调用链路                         │
│  3. 触发条件是什么？      → 确定输入和环境                       │
│  4. 正确应该是什么？      → 对比预期行为                         │
└─────────────────────────────────────────────────────────────────┘
```

#### 根因分析技术

##### 代码审查
```markdown
## 代码审查清单

### 检查项
- [ ] 变量是否正确初始化？
- [ ] 条件判断逻辑是否正确？
- [ ] 循环边界是否正确？
- [ ] 资源是否正确释放？
- [ ] 异常是否被正确处理？
```

##### 日志分析
```markdown
## 日志分析要点

### 关键词搜索
- error, exception, failed
- null, undefined, nil
- timeout, refused
- denied, unauthorized

### 时间线分析
- 按时间顺序排列事件
- 识别异常点前后的事件
- 寻找模式或相关性
```

##### 数据追踪
```markdown
## 数据追踪方法

### 输入追踪
- 数据从哪里来？
- 经过了哪些处理？
- 最终变成了什么？

### 状态追踪
- 对象状态是否正确？
- 是否有竞态条件？
- 是否存在状态不一致？
```

## 刑部调试原则

### 核心原则
1. **不问责任，只找根因**：刑部是诊断部门，不是审判部门
2. **精准定位**：提供足够的上下文帮助五监定位修复
3. **可复现**：确保问题可以被稳定复现
4. **不过界**：不替五监写修复代码，只提要求

### 刑部不应做的事
- ❌ 替五监写修复代码
- ❌ 随意猜测根因
- ❌ 隐瞒问题或误判归责
- ❌ 超范围调试（非职责内的代码）
