# 代码审查技能
工部（Engineering Ministry）必须执行的标准代码审查流程。

## 审查范围

### 1. 代码质量
- [ ] **可读性**：变量/函数命名清晰，逻辑直观
- [ ] **简洁性**：避免不必要的复杂性
- [ ] **一致性**：遵循项目编码规范
- [ ] **模块化**：职责单一，接口清晰

### 2. 逻辑正确性
- [ ] **边界条件**：空值、零值、极值处理
- [ ] **错误处理**：异常情况有兜底
- [ ] **并发安全**：多线程场景正确处理
- [ ] **资源管理**：内存/文件/连接正确释放

### 3. 性能考量
- [ ] **算法复杂度**：避免 O(n²) 可优化场景
- [ ] **数据库查询**：N+1 问题，避免全表扫描
- [ ] **缓存策略**：合理利用缓存减少计算
- [ ] **懒加载**：按需加载大资源

### 4. 安全检查
- [ ] **输入验证**：所有外部输入经过验证
- [ ] **SQL 注入**：使用参数化查询
- [ ] **XSS 防护**：输出转义
- [ ] **敏感数据**：不硬编码密钥/凭据

### 5. 测试覆盖
- [ ] **新增代码**：是否有对应测试
- [ ] **边界测试**：边界条件覆盖
- [ ] **异常测试**：错误路径测试

## 审查流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   PREPARE   │───►│   REVIEW    │───►│   COMMENT   │───►│   RESOLVE   │
│   准备      │    │   审查      │    │   反馈      │    │   确认      │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### Phase 1: 准备审查
1. 理解变更目的和背景
2. 查看相关设计文档
3. 运行本地测试确保基础通过
4. 检查变更范围

### Phase 2: 执行审查
1. 自顶向下阅读代码
2. 关注业务逻辑而非风格
3. 记录问题和建议
4. 识别亮点和优秀实践

### Phase 3: 撰写反馈
1. 分类问题严重程度
2. 提供具体改进建议
3. 解释问题原因和风险
4. 标注"建议"vs"必须修改"

### 问题严重程度

| 级别 | 标记 | 说明 | 处理要求 |
|------|------|------|----------|
| 阻塞 | 🔴 blocker | 必须修复才能合并 | 阻塞合并 |
| 严重 | 🟠 major | 强烈建议修复 | 需要讨论 |
| 轻微 | 🟡 minor | 建议改进 | 可后续优化 |
| 赞誉 | 🟢 praise | 优秀实践值得推广 | 表扬鼓励 |

### Phase 4: 跟进确认
1. 等待开发者修复
2. 验证修复是否正确
3. 确认所有问题已解决
4. 批准合并或提出最终意见

## 审查检查清单

### 必查项目
```markdown
## 代码审查清单

### 功能正确性
- [ ] 代码逻辑是否正确实现需求？
- [ ] 是否有遗漏的边界情况？
- [ ] 错误处理是否完善？

### 代码质量
- [ ] 函数/类是否单一职责？
- [ ] 变量命名是否有意义？
- [ ] 是否有重复代码可以提取？
- [ ] 注释是否必要且准确？

### 性能与安全
- [ ] 是否有性能隐患？
- [ ] 是否符合安全最佳实践？
- [ ] 是否正确处理敏感数据？

### 测试
- [ ] 新功能是否有测试？
- [ ] 测试是否覆盖关键路径？
- [ ] 测试是否可维护？
```

## 最佳实践

### 给开发者的建议
1. **保持变更小巧**：每次 PR 少于 400 行
2. **自审查**：提交前自己先审查一遍
3. **写好描述**：说明 Why 而非 What
4. **响应及时**：尽快处理审查反馈

### 给审查者的建议
1. **及时审查**：收到请求后 24 小时内完成
2. **建设性反馈**：指出问题时提供解决方案
3. **区分主次**：阻塞问题明确标注
4. **考虑情境**：理解业务背景和约束

### 特殊场景处理

#### 大型变更
- 分多次 PR 逐步合并
- 先合并基础设施，后合并业务逻辑
- 关键路径优先审查

#### 紧急修复
- 标注 "hotfix" 或 "urgent"
- 简化为最小可用修复
- 事后补充测试和文档

#### 架构重构
- 提供设计文档和对比
- 拆分为多个独立步骤
- 确保向后兼容
