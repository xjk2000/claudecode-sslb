# 测试驱动开发技能
兵部（Testing Ministry）必须执行的 RED-GREEN 循环流程。

**核心原则**：兵部只负责测试验证，发现失败后打回给五监修复。

## 职责划分

```
┌─────────────────────────────────────────────────────────────────┐
│                         三省六部职责                              │
├─────────────────────────────────────────────────────────────────┤
│  兵部（测试部）                                                 │
│  ├── 接收尚书令任务                                             │
│  ├── 分析测试需求，编写测试用例                                  │
│  ├── 运行测试，验证结果                                          │
│  └── 测试失败 → 打回给五监对应部门                               │
├─────────────────────────────────────────────────────────────────┤
│  五监（实现部）                                                 │
│  ├── 将作监：核心业务逻辑                                       │
│  ├── 少府监：前端交互逻辑                                       │
│  ├── 军器监：安全相关代码                                       │
│  ├── 都水监：数据调整相关                                       │
│  └── 国子监：框架/架构相关                                      │
└─────────────────────────────────────────────────────────────────┘
```

## RED-GREEN 循环流程

```
┌─────────┐         ┌─────────┐         ┌─────────┐
│   RED   │────────►│  打回   │────────►│  GREEN  │
│  写测试  │         │  五监   │         │  验证   │
│ (失败)  │         │  修复   │         │ (通过)  │
└─────────┘         └─────────┘         └────┬────┘
                                             │
                                             ▼
                                      ┌─────────────┐
                                      │   下一轮    │
                                      │ 继续分析    │
                                      └─────────────┘
```

### 兵部TDD标准流程

#### Step 1: 接收尚书令任务
- 明确需要测试的功能点
- 分析涉及的五监部门
- 制定测试计划

#### Step 2: RED - 编写测试用例
```markdown
## 测试用例编写

### 分析功能点
- 确定被测代码属于哪个五监部门
- 识别输入、输出、边界条件
- 设计测试用例覆盖所有场景

### 编写测试代码
- 遵循项目测试规范
- 包含 Arrange-Act-Assert 结构
- 添加清晰的测试描述
```

#### Step 3: 运行测试，确认失败
- 执行测试命令
- 记录失败信息和错误堆栈
- 确认失败原因

#### Step 4: 编写《打回敕令》
兵部测试失败后，必须填写打回敕令，打回给五监：

```markdown
# 兵部打回敕令

**敕令编号**: BTD-YYYYMMDD-XXX
**日期**: YYYY-MM-DD
**兵部呈**: 尚书令
**主送**: [五监对应部门]
**抄送**: 尚书令、门下省

## 一、测试任务
[描述本次测试的任务]

## 二、测试环境
- 测试文件: [path/to/test]
- 被测模块: [module/path]
- 相关五监: [将作监/少府监/军器监/都水监/国子监]

## 三、测试结果

### RED阶段 - 测试失败
- 状态: 🔴 失败
- 测试命令: `npm test -- --testPathPattern=xxx`
- 退出码: [code]

### 失败详情
```
[粘贴失败信息和错误堆栈]
```

## 四、失败原因分析
[分析失败原因，明确责任部门]

## 五、打回要求
[明确需要修复的内容和标准]

## 六、回复期限
[设置合理的回复期限]

---
兵部主事（签字）: ___________
```

#### Step 5: 五监接收并修复
五监收到打回敕令后：
1. 分析失败原因
2. 修复代码
3. 回复兵部

#### Step 6: GREEN - 兵部验证
兵部收到修复后：
1. 重新运行测试
2. 确认测试通过
3. 汇报尚书令

## 测试覆盖率要求

### 最低覆盖率
- 语句覆盖率: ≥ 80%
- 分支覆盖率: ≥ 70%
- 函数覆盖率: ≥ 90%

### 覆盖率报告生成
```bash
# 生成覆盖率报告
npm test -- --coverage

# 查看覆盖率
open coverage/lcov-report/index.html
```

## 测试分类

### 按职责分类
| 测试类型 | 负责部门 | 执行时机 |
|----------|----------|----------|
| 单元测试 | 五监自行 | 开发时 |
| 集成测试 | 兵部执行 | 提测后 |
| 端到端测试 | 兵部执行 | 发布前 |

### 按场景分类
```markdown
## 测试场景

### 正常路径
- 验证标准输入产生预期输出

### 边界条件
- 空值、零值、最大值
- Unicode字符、特殊符号
- 极端数据量

### 异常处理
- 无效输入
- 网络失败
- 资源不可用
```

## 常见问题处理

### 测试失败打回标准
| 情况 | 处理 |
|------|------|
| 编译错误 | 打回给五监（编译失败是实现问题） |
| 断言失败 | 打回给五监（实现与预期不符） |
| 环境问题 | 兵部自行解决或上报尚书令 |
| 测试代码bug | 兵部自行修复 |

### 拒绝打回情况
兵部不应打回测试本身的问题给五监，例如：
- 测试用例编写错误
- 测试数据准备不当
- 测试环境配置错误
